# Backend Implementation Plan - Remaining Tasks

## الهدف
إكمال الباكند بحيث يدعم دورة الخام الجديدة بشكل صحيح وبدون أي خصم مبكر أو فقد في التتبع.

## ما تم تنفيذه بالفعل
### 1) أوامر التشغيل
- فصل الحجز عن الاستهلاك.
- إنشاء أمر التشغيل الآن يعمل كـ `reserved`.
- `dispatch` هو نقطة الخصم الفعلية من الخام.
- `cancel` يحرر الكمية إذا كانت صُرفت بالفعل.

### 2) الخام المتاح
- تحديث حساب `available_qty` ليأخذ في الاعتبار:
  - المحجوز
  - المصروف
  - المستخدم فعليًا
- تحديث queries الخام المتاح حتى لا تُعرض الكميات المحجوزة كمتاحة.

### 3) الفرزة
- جعل `production_order_id` إلزاميًا في خطوط الفرزة.
- عند `post` للفرزة يتم إنشاء حركات مخزون للخام والمنتج النهائي والهالك.

### 4) البالتيه
- إضافة server-side validation تمنع تجاوز cartons المتاحة من سطر الفرزة.

### 5) الـ API / resources
- إضافة labels للحالات الجديدة في الخام وأوامر التشغيل.

## المتبقي على مستوى الـ Backend

### أولوية عالية

#### 1) تثبيت حركة المخزون النهائية
**الهدف:** ضمان أن كل حركة خام/منتج/هالك تُسجَّل بشكل كامل ومترابط.

**المطلوب:**
- مراجعة `StockTransaction` بحيث تغطي كل الحالات المطلوبة:
  - خام مستهلك
  - منتج نهائي داخل
  - هالك خارج
- التأكد أن كل `post` للفرزة يكتب القيود المطلوبة بدون تكرار.
- التأكد أن أي إعادة post أو retry لا تعمل duplication.

**الحالة:** مكتمل

**ما تم عمله:**
- إضافة `sort_record_line_id` إلى `stock_transactions`.
- إنشاء unique constraint على `(sort_record_line_id, type, reason)`.
- ربط كل transaction بسطر الفرزة نفسه داخل `SortRecordService::post`.
- منع إعادة `post` لو الفرزة لم تعد في حالة `draft`.

**معيار القبول:**
- كل sort record posted ينتج عنه الحركات الصحيحة مرة واحدة فقط.
- لا يوجد سجل خام أو فرزة مكتمل بدون trace واضح في `stock_transactions`.

#### 2) Hardening لأوامر التشغيل
**الهدف:** منع أي تلاعب أو حالة غير متوقعة في الحجز والصرف والإلغاء.

**المطلوب:**
- التأكد أن `dispatch` لا يعمل على أمر ملغي.
- التأكد أن `cancel` يعكس الصرف فقط إذا كان الأمر بالفعل dispatched.
- التأكد أن `delete` يحرر أي كمية مرتبطة قبل الحذف.
- التأكد من عدم تجاوز `used_quantity` أو `available_qty` في أي حالة.

**معيار القبول:**
- لا يمكن dispatch/cancel/delete في حالة تكسر رصيد الخام.
- كل transition يغيّر حالة الخام بشكل صحيح.

**الحالة:** مكتمل — تم تنفيذ قفل الصفوف، تحقق الكمية قبل الصرف، والتعامل مع حالات idempotency.

#### 3) Backend validation للفرزة
**الهدف:** منع حفظ بيانات فرزة غير قابلة للتتبع أو غير متوازنة.

**المطلوب:**
- التأكد أن كل line يحتوي `production_order_id`.
- التأكد أن `line_total` محسوب بشكل متسق.
- إضافة validation أو guard لو إجمالي الهالك/المنتج غير منطقي.
- التأكد من أن `post` لا يعمل إذا لم توجد lines.

**معيار القبول:**
- أي sort record غير مكتمل يتم رفضه قبل الحفظ أو قبل post.
- لا يتم إنشاء stock movements إلا بعد نجاح كل validation.

**الحالة:** مكتمل — تم تنفيذ التحقق التالي في `SortRecordService` وجرى إضافة اختبارات:
- إلزام وجود خطوط عند الإنشاء/التحديث ورفض السجل إذا لم توجد خطوط.
- إجبار وجود `production_order_id` لكل سطر ورفض أي سطر ناقص.
- حساب `line_total` برمجياً من القيم المدخلة (`grade_a/b/c + waste + returned`).
- منع `post` إذا كان أي سطر يفتقد `productionOrder` أو `rawReceipt`، أو إذا كانت أجزاء السطر لا تتطابق مع `line_total`.
- إضافة اختبارات وظيفية جديدة في `tests/Feature/SortRecordValidationTest.php` لتغطية سيناريوهات الإدخال غير الصالح.

**ملاحظة تشغيل الاختبارات:** حاولت تشغيل الاختبارات لكن التنفيذ فشل بسبب إعدادات قاعدة بيانات الاختبار (`packing_test`) — خطأ اتصال MySQL (Access denied). يلزم توفير بيانات اعتماد اختبار صالحة أو تفعيل PDO SQLite لتشغيل الاختبارات محلياً.

### أولوية متوسطة

#### 4) تقارير وحالات الخام
**الهدف:** تجهيز الـ API والتقارير لتعرض الحالات الجديدة بوضوح.

**المطلوب:**
- التأكد أن resources تعرض:
  - `available_qty`
  - `status_label`
- إضافة/مراجعة أي endpoints أو exports تحتاج:
  - الخام المحجوز
  - الخام المصروف
  - الخام المستهلك
- تجهيز أي summary data للداشبورد إن كان الباكند هو مصدرها.

**معيار القبول:**
- أي شاشة UI تعتمد على backend تستطيع التمييز بين `reserved` و `dispatched` و `consumed`.
- التقارير لا تُظهر الخام كمتاح وهو محجوز.

#### 5) waste monitoring
**الهدف:** دعم أي تنبيه أو تقرير هالك من الباكند.

**المطلوب:**
- التأكد أن `has_waste_alert` يحسب بدقة.
- إضافة أي endpoints needed لعرض الهالك ونسبته.
- التأكد أن القيم الناتجة ثابتة بعد post.

**معيار القبول:**
- يمكن للواجهة الاعتماد على قيمة واحدة صحيحة للـ waste alert.
- أي record يتجاوز threshold يظهر عليه تنبيه backend-side.

### أولوية لاحقة

#### 6) cleanup للـ legacy flows
**الهدف:** إزالة أي سلوك قديم يفترض أن الخصم يحدث عند الإنشاء.

**المطلوب:**
- مراجعة أي service/repository ما زال يخصم raw receipt مباشرة عند create.
- مراجعة أي controller أو action قديم يؤثر على نفس lifecycle.
- توثيق السلوك الجديد في الـ API أو README أو docs الخاصة بالـ backend.

**معيار القبول:**
- لا يوجد path قديم يعيدنا للخصم المبكر.
- السلوك موحد في كل endpoints.

#### 7) استقرار الاختبارات
**الهدف:** جعل التغييرات قابلة للثقة والرجوع لها.

**المطلوب:**
- تشغيل اختبارات lifecycle الخاصة بـ production orders.
- تشغيل اختبارات sort record و pallet validation.
- إضافة أي test ناقص على الحالات الجديدة.
- إصلاح أي failure يظهر في بيئة الاختبار عند توفر قاعدة بيانات مناسبة.

**معيار القبول:**
- اختبارات الـ backend المهمة تمر بثبات.
- أي تغيير في lifecycle يكون مغطى باختبار.

## ترتيب التنفيذ المقترح
### المرحلة 1 - تأمين السلوك الأساسي
1. تثبيت `StockTransaction` lifecycle.
2. مراجعة dispatch/cancel/delete في production orders.
3. تأكيد validation الخاص بالفرزة.

### المرحلة 2 - التعريض والتقارير
4. مراجعة resources والـ exports للحالات الجديدة.
5. إضافة/تثبيت waste alerts والتقارير.
6. تجهيز أي summary endpoints يحتاجها الـ UI.

### المرحلة 3 - تنظيف السلوك القديم
7. إزالة/تحديث أي flow قديم يخصم عند الإنشاء.
8. توثيق السلوك الجديد.
9. تثبيت الاختبارات النهائية.

## قائمة تنفيذ سريعة
- [ ] مراجعة `SortRecordService::post` والتأكد من عدم تكرار stock transactions.
- [ ] مراجعة `ProductionOrderRepository::dispatch` و `cancel` و `delete`.
- [ ] مراجعة `RawReceipt::availableQty()` للتأكد من الصيغة النهائية.
- [ ] مراجعة الموارد والـ exports للحالات الجديدة.
- [ ] تثبيت أي endpoint للداشبورد أو التقارير المطلوبة.
- [ ] تشغيل اختبارات الـ backend بعد ضبط قاعدة البيانات.
- [ ] توثيق السلوك النهائي للـ API.

## تعريف النجاح
الـ backend يعتبر مكتملًا عندما:
- لا يوجد خصم خام عند إنشاء أمر التشغيل.
- كل dispatch/cancel/post يترك trace واضح.
- الخام المتاح والحجز والصرف والهالك كلها محسوبة بشكل واحد.
- الـ API يمد الواجهة بكل ما تحتاجه بدون inference خاطئ.
- الاختبارات الأساسية تمر.
