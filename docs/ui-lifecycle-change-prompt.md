# Prompt: تعديلات الـ UI المطلوبة لدورة الخام والفرزة

أريد منك تنفيذ تعديلات الـ UI التالية في مشروع Laravel/Frontend الحالي بحيث يتوافق تمامًا مع السيناريو الجديد لدورة الخام:

## الهدف العام
- فصل الحجز عن الاستهلاك في أوامر التشغيل.
- إظهار الخام المتاح فعليًا فقط عند الاختيار.
- إجبار الفرزة على الارتباط بأمر تشغيل.
- منع pallets من تجاوز cartons المتاحة.
- عرض الحالات الجديدة بوضوح في القوائم والتفاصيل والتقارير.
- جعل المستخدم يرى الفرق بين:
  - خام كلي
  - خام متاح
  - خام محجوز
  - خام مصروف
  - خام مستهلك

## المطلوب تنفيذه في الـ UI

### 1) شاشة Raw Receipts List
- أضف عمود `available_qty` بوضوح.
- أضف badge للحالة يوضح:
  - `in_stock` = في المخزون
  - `reserved` = محجوز
  - `dispatched` = مصروف
  - `consumed` = مستهلك
- لو `available_qty = 0`، اعرض الصف كغير قابل للاستخدام.
- امنع اختيار الخام المنتهي في أي dropdown أو selector.
- بعد أي عملية حفظ أو dispatch أو cancel، اعمل refresh للقائمة.

### 2) شاشة Production Order Form
- اختار الخام من قائمة الخام المتاح فقط.
- اعرض `available_qty` داخل نتائج البحث أو dropdown.
- امنع حفظ أمر التشغيل إذا كانت `target_qty_kg` أكبر من `available_qty`.
- بعد إنشاء أمر التشغيل، اعرض الحالة `reserved` بوضوح.
- بعد dispatch، حدّث الحالة إلى `dispatched` مباشرة.
- بعد cancel، اعرض سبب الإلغاء إن وجد.

### 3) شاشة Production Orders List
- أضف badges للحالات الجديدة، خاصة `reserved` و `dispatched`.
- أظهر الكمية الخام المتبقية أو المتاحة من الخام المرتبط.
- فعّل الأزرار حسب الحالة فقط:
  - Draft: تعديل / حذف
  - Reserved: dispatch / cancel
  - Dispatched: حسب منطق النظام
  - Cancelled: read-only
- أي فلتر للحالة يجب أن يدعم الحالات الجديدة.

### 4) شاشة Sort Record Form
- اجعل `production_order_id` إلزاميًا في كل line.
- لا تسمح بإرسال line بدون أمر تشغيل.
- اعرض live totals لكل line:
  - A + B + C + waste + returned
- اعرض total للفرزة بشكل لحظي.
- لو `total_sort = 0`، امنع الحفظ أو الترحيل.
- لو نسبة الهالك مرتفعة، اعرض warning قبل post.

### 5) شاشة Post Sort Record
- بعد الترحيل، اقفل التعديل والحذف.
- اعرض ملخص واضح بعد post:
  - الخام المستهلك
  - المنتج النهائي
  - الهالك
- حدّث status إلى `posted` مباشرة في الواجهة.
- حدّث أي counters أو dashboard cards مرتبطة بالفرز.

### 6) شاشة Pallet Modal / Pallet Form
- اعتمد على `available_order_id` أو سطر فرزة مرحّلة بدل الإدخال اليدوي.
- اعرض cartons المتاحة من السطر قبل الحفظ.
- امنع إدخال `cartons_count` أكبر من المتاح.
- لو السطر غير مرتبط بأمر تشغيل أو لم يكن posted، لا تسمح باختياره.
- لا تعتمد على حسابات الواجهة وحدها؛ اعرض قيمة السيرفر وطبّق validation client-side أيضًا.

### 7) شاشة Pallet Details
- اعرض الربط الحقيقي بين pallet و sort line و production order.
- أوضح مصدر pallet من أي sort record line.
- أظهر الحالة الحالية بوضوح.

### 8) Dashboard
- أضف cards أو summary metrics للآتي:
  - الخام المحجوز
  - الخام المصروف
  - الفرزات المرحّلة
  - الهالك اليومي
- أظهر alert إذا `has_waste_alert = true`.
- حدّث الأرقام بعد أي عملية create / dispatch / post / cancel.

### 9) Reports
- عدّل التقارير بحيث تعرض الحالات الجديدة بدل الاعتماد على `in_stock` فقط.
- أظهر الهالك ونسبة الفقد إن أمكن.
- تأكد أن تقارير الخام والإنتاج متوافقة مع التغييرات الجديدة.

## قواعد مهمة جدًا
- لا تعتمد على `quantity_kg` وحدها في أي اختيار تشغيلي.
- `available_qty` هو المصدر الصحيح للاستخدام في التشغيل.
- لا تسمح للواجهة بإنشاء pallet بكمية cartons أكبر من المتاح.
- لا تسمح للواجهة بحفظ Sort Record line بدون `production_order_id`.
- لا تعرض الخام المحجوز كخام متاح للاختيار.
- حافظ على الرسائل العربية المتوافقة مع الـ backend.
- بعد أي نجاح في create/dispatch/post/cancel، اعمل refresh للبيانات المتأثرة.

## Acceptance Criteria
- لا يمكن إنشاء أمر تشغيل على خام غير متاح.
- لا يمكن حفظ فرزة بدون `production_order_id`.
- لا يمكن حفظ pallet بكمية cartons أكبر من المتاح.
- تظهر الحالات الجديدة في كل القوائم والتفاصيل بشكل واضح.
- يظهر الفرق بين raw total و raw available و raw reserved و raw dispatched بوضوح للمستخدم.

## مخرجات مطلوبة منك
- حدّد الملفات/المكونات التي ستتغير في الـ frontend.
- نفّذ التعديلات مع أقل تغييرات ممكنة.
- حدّث أي hooks أو API modules أو validators مرتبطة.
- تأكد أن الواجهة تتصرف بشكل صحيح بعد كل عملية حفظ.
